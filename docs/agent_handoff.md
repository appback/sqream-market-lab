# Agent Handoff Guide

This document is the first read for a new agent taking over SQream Market Lab.
Keep it current whenever runtime behavior, schedules, strategy policy, reporting, or infrastructure changes.

## Operating Goal

Run paper-trading experiments for US stocks while keeping the production path conservative:

1. collect intraday market data,
2. stage it as Parquet,
3. load and analyze with SQream,
4. detect strategy targets,
5. manage paper trades,
6. report market open, target detection, paper buy/sell, errors, and daily close.

The system is not live trading yet. Treat all outputs as paper-trading evidence, not a live order recommendation.

## SQream Access

Use the manager/picker endpoint, not direct worker endpoints:

- host: `192.168.0.26`
- database: `master`
- manager/picker ports: `3105`, `3108`
- worker ports: `5000` through `5004`
- user: `sqream`
- password: stored/known operational credential, do not print in reports
- JDBC service examples:
  - ingest: `service=ingest`
  - analysis: `service=analysis`
  - runtime detection: `service=sqream`

Current worker grouping is documented in `docs/sqream_worker_grouping.md`.

## SQream Worker Policy

Current policy is `2 ingest / 2 analysis / 1 sqream`:

- `ingest`: Parquet staging load only,
- `analysis`: latest rebuild, market regime, strategy SQL,
- `sqream`: lightweight runtime detection and paper-trading SQL.

Do not route every query through one generic service. Preserve service separation so report/detection queries are not blocked by heavier analysis jobs.

## Ingestion Policy

CSV is not allowed for market-data ingestion.

Current accepted path:

1. collect bars,
2. write Parquet staging file,
3. load through SQream Parquet FDW/foreign table flow,
4. insert into SQream raw/latest tables.

If a new format is proposed, benchmark it against Parquet first and document the result in `docs/ingestion_format_evaluation.md`.

## Reporting Policy

Product reporting uses `REPORTBOT_TOKEN` through `report_notifier.py`.
Never print or persist the token.

Report these events:

- market preflight result,
- market open,
- market close and daily settlement,
- runtime error,
- detected target,
- paper buy,
- paper sell,
- daily PnL summary.

Messages must be readable:

- include display name and internal strategy when useful,
- include symbol and company name if available,
- include action: detect / buy / sell / settle,
- include price, target, stop, PnL, regime, time bucket,
- avoid one-line packed key/value dumps for daily reports.

## Common Failure Modes

1. SQream down at market open.
   - Check ports `3105`, `3108`, `5000~5004`.
   - Run preflight.
   - Restart SQream only with the correct account and service layout.

2. Telegram report test works but runtime reports do not.
   - Check cron/runtime environment.
   - Confirm `report_notifier.py` uses the project secret path.
   - Run the actual runtime script path, not just a standalone token test.

3. Duplicate reports.
   - Check whether both RemoteAgent helper delivery and product bot delivery are active for the same event.
   - Product reports should use `report_notifier.py`; conversational replies should stay in normal RemoteAgent text.

4. Misleading strategy totals.
   - Separate recent, all-time, active-only, and current-policy-filtered results.
   - For D1, do not mix pre-policy non-opening trades into current-policy conclusions.

5. SQream bottleneck.
   - Do not add ingest workers blindly.
   - First check `sqream_lock_wait`, load time, latest rebuild time, and service routing.
   - Keep ingest append-only where possible and move heavier rebuilds to analysis.
